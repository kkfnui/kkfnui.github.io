svn分支开发流程说明

发表于 2014-01-03   |  

工作中我们使用svn来管理源码。之前基于svn来开发需求的基本流程：

  1. 开发在本地将需求完成
  2. 将代码修改提交到svn，由编译机统一编译
  3. 统一编译后，将产品提交给测试
  4. 开始测试产品

这样的使用方法会有以下几个问题：

  1. 不能及时的将代码提交到svn上，存在代码丢失的风险

  2. 需求内bug的修复，提交svn产生噪音

    上面开发的基本流程，在最理想的状态下是没有问题的。事实上，步骤2，4都会产生问题,造成主干上的垃圾日志就变多。

    • 提交时忘记先添加某个新加的文件(开发本地编译没有问题，但是编译机无发通过)
    • 测试发现问题，这个时候必须修复问题吧，又得提交一次

  3. 不方便bug定位

    这点也是因为上面说的日志噪音产生的。想象下产品上线后，发现一个bug，我们需要定位是哪个需求引入的。
    一个需求的svn提交记录很多，到底改了哪些东西就不是那么容易确定了。

svn使用流程

对于上面提出的问题，解决方案会有多个。可以要求开发人员提交svn时不能出现低级错误，开发需求不能出现bug。好吧，这个是开玩笑的了。从现在的经验看来，这些问题都无法避免。错误既然无法避免，那么就修改流程允许错误的出现，同时不会带来坏的影响。

下面就是我关于svn使用流程的一个想法：

  1. 一个需求必须创建一个独立的svn分支
  2. 开发和测试这个过程，都是基于这个svn分支的
  3. 测试确认最终没有问题后，合并到主干上

在理想情况下，主干上的每次修改都是稳定的、可发布的。

同上面分析以前的流程一样，我们同样不能将整个流程看的太顺利。

  1. 创建svn分支。

    不会创建svn分支？不能创建svn分支？如果在这里出现了问题，影响将会是最小的

  2. 开发和测试的迭代

    这个过程是允许出错的

  3. 合并需求到主干

    这个过程很重要，这就相当于一个产品的发布。在没有修改冲突的时候，我们可以直接使用svn的merge操作。比较麻烦的是有冲突，需要相关人员一起来合并代码了。

问题

  1. 每个需求都创建一个分支是否有必要？

    有必要。

    如果不是一个需求一个分支，那么两种可能：多个需求多个分支和一个需求多个分支（有病的感觉，纯粹是分析问题为了保证完整性）；多个需求一个分支，这个挺正常的。

    先来分析下最后一种情况，说下这样做不太合适的地方。合并一个分支到主干发布，这个分支包括了需求1、2、3。现在产品发布出去了，结果出现了bug（性能问题）。现在我们要分析是那个改动引入的。分析定位问题要做到，最终问题可以被，所以越精确越好。这种情况逐渐缩小搜索问题可能出现的范围是一个最好的解决方案。即需要定位哪个需求引入的，那么要从一次提交中拆分三个不同的需求。没有做过需求的人根本不知道怎么拆分。

    每个需求一个分支除了能解决上面的问题，还允许并行开发。

    最后，我有一个目标，就是在主干的日志上，可以一目了然的看清楚在什么时间发布了什么改动。

  2. 很简单的一个需求，创建一个分支，最后还需要合并代码是不是过于麻烦了？

    在回答这个问题前，我希望我们对于第一个问题已经达成共识。

    简单的定义，这是一个比较泛的东西，不容易界定。这里就给它一个定义：与其他需求在代码的修改上不存在冲突。现在这个问题的关键字就是麻烦了。我不认为这是一个麻烦的操作，不存在冲突，合并代码这个操作工具就可以帮助完成了。

工具的使用

工具不是我们的目标，但是工具可以帮助我们实现目标，影响我们的思路。

下面我说下我对svn分支操作的理解。工作中都是使用Tortoise SVN客户端的，下面就使用它来说明下相关的操作。

  • 如果svn基本操作不会的可以先参考：http://blog.csdn.net/dily3825002/article/details/6717076
  • 更多更具体的操作可以参考：http://tortoisesvn.net/docs/nightly/TortoiseSVN_zh_CN/index.html

按照上面说的流程，需要创建分支、在分支上开发、合并分支。

创建分支比较简单，可以参考上面的两个链接。

创建完成分支之后，需要在分支上开发。以前我比较笨，就从svn的上checkout分支下来。其实不需要这么麻烦，svn有个switch命令，可以改变本地路径对应的svn url。给人的感觉，创建分支是和没有创建分支一样。详细的介绍：http://www.cnblogs.com/dabaopku/archive/2011/05/21/2052820.html

分支合并有两种情况，没有冲突和有冲突。

Phrase Emphasis

*italic*   **bold**
_italic_   __bold__

Links

Inline:

An [example](http://url.com/ "Title")

Reference-style labels (titles are optional):

An [example][id]. Then, anywhere
else in the doc, define the link:

  [id]: http://example.com/  "Title"

Images

Inline (titles are optional):

![alt text](/path/img.jpg "Title")

Reference-style:

![alt text][id]

[id]: /url/to/img.jpg "Title"

Headers

Setext-style:

Header 1
========

Header 2
--------

atx-style (closing #’s are optional):

# Header 1 #

## Header 2 ##

###### Header 6

Lists

Ordered, without paragraphs:

1.  Foo
2.  Bar

Unordered, with paragraphs:

*   A list item.

    With multiple paragraphs.

*   Bar

You can nest them:

*   Abacus
    * answer
*   Bubbles
    1.  bunk
    2.  bupkis
        * BELITTLER
    3. burper
*   Cunning

Blockquotes

> Email-style angle brackets
> are used for blockquotes.

> > And, they can be nested.

> #### Headers in blockquotes
> 
> * You can quote a list.
> * Etc.

Code Spans

`<code>` spans are delimited
by backticks.

You can include literal backticks
like `` `this` ``.

Preformatted Code Blocks

Indent every line of a code block by at least 4 spaces or 1 tab.

This is a normal paragraph.

    This is a preformatted
    code block.

Horizontal Rules

Three or more dashes or asterisks:

---

* * *

- - - -

Manual Line Breaks

End a line with two or more spaces:

Roses are red,   
Violets are blue.

解析markdown的想法

发表于 2013-12-27   |  

前因

这几天用Markdown写东西越写越high了。写文主要的是内容，但是Markdown的体验真的很不错。网上有各种Markdown的编辑器，各有各的特点，功能基本满足我的需求。

StackEdit 这个在线编辑器功能很强大，用户体验很好,这篇文档就在这上边编写完成的。它提供了和google、dropbox以及github的互操作，对我来说的功能已经溢出了。

前几天使用的是马克飞象 这款编辑器写的文档，可以和Evernote打通，也非常棒。比较两者，发现各自对Markdown的语法支持有差异。

使用两者之后的感受，各有各的好处吧。但是这种面向大众使用产品，个人的操作效率肯定不是最高的。比如，我特别想通过StackEdit一键发布博文，保存到GitHub的路径可以自动确定。有种声音，改变它，让它符合自己的操作习惯。

上面提到的StackEdit编辑器是开源的，如果想增加特化的功能，可以自己修改源码实现。源码地址：https://github.com/benweet/stackedit

从满足自己的需求角度出发，我应该基于StackEdit自己再做二次开发的。可是总有一种想法驱赶我:”你应该自己造个轮子解析Markdown”。
这两个事相差的有点远了。有些时候，也不知道自己为什么会这样想。

应该就是我能这个道理吧。

一点想法

词法解析这块完全没有经验。

印象中需要使用到树和状态机吧。大学里面没有好好学习数据结构，学了的也还给老师了。所以这块肯定是要重现学习的。状态机倒也还算实践过，但却也没有说能轻松搞定的勇气。

涉及到字符串编码的问题，这个倒还比较熟悉。却也没有真正研究过，比如在内存解析时使用utf8编码，怎么遍历字符和字符匹配呢？

总体来说，是零基础开始做了。

计划

这块肯定已经有比较成熟的解析器。应该先借鉴下现有的实现，怎么也得踩下别人的肩膀。另外现在已经可以预见的知识也要准备下：

  • 树数据结构算法
  • 状态机
  • 字符串编码
  • Markdown的语法

最后一句，”善始者实繁，克终者盖寡”。算是激励自己，也为自己找个台阶。

补充：突然想到自己做事不从需求出发，应该是被人洗脑了，看大神言论看多了。算了，这次认栽了！

工作后第一次游泳

发表于 2012-08-09   |  

想来自己已经很长时间没有游泳了。在读大学之前，每年放暑假的时候我都会到家附近的一条小河去游泳。和邻居的玩伴一起去游泳的日子已经只能成为回忆了。

大学的时候对游泳最深刻的记忆便是在学校的游泳馆里游泳。那是大三暑假留校为网站“改版”了。在学校里面游泳的时候，有个现象让我感到诧异。体力并非有我好的人，可游泳却比我游的远。那时候也没有细想这个。这点来看我对周围的事物的敏感度，以及了解事物的本质的欲望都不强。

近些时间，部门里面每周都有游泳的机会。可是由于之前一直忙着工作，或是去打羽毛球的原因，一直没有去游泳。今天恰逢空闲，便和同事一起出来游泳。

在路上就和同事在谈论游泳。当然对于我们这些游泳技术不怎么地人来讲，谈话的过程中就自然会涉及到“蛙泳”。之前我以为所有的游泳方式都是会非常消耗体力的，在一番交谈之后。蛙泳是一种非常不耗体力的印象，已经深深的刻入我的脑海。

自然而然，到了泳池我便开始尝试学习蛙泳了。对于我之前已经会“狗爬式”游泳的人来说，会碰到以下的问题：

  1. 如何换气。 之前“狗爬式”的泳姿，头是一直付出水面的，不需要换气。
  2. 蛙泳的动作。 我之前游泳都是很随意的，有力就使劲划，没力了就慢慢来，没有规定的动作。

在泳池中游泳我也是要解决这两个问题。幸运的是我之前就会游泳，所以一开始就可以在水中去练习这些动作了。

在经过了一两个小时的练习之后，算是对蛙泳有点门路了。解决上面的两个问题就是：

  1. 在浮出水面之前开始吐气，等头伸出水面的时候就吸气。
  2. 动作还是比较复杂的，可以参考别人总结的：

划手腿不动，
收手再收腿，
先伸手臂再蹬腿，
并拢伸直漂一会。

游泳游到最后有点感觉了，同事就准备回去了，今天还是游的未尽兴。

最后想起来了，我还有一个比较麻烦的问题：鼻子容易进水。

捣腾隐藏任务栏的小工具

发表于 2012-03-10   |  

平时写工作的时候，会开很多程序：记事本、浏览器、Explorer、IDE、QQ、RTX等。
在写程序的时候看到这样的任务栏，觉得非常乱呀。
有段时间我试过将任务栏设置成自动隐藏的，但是鼠标一不小心划到任务栏上，它就弹出来了，就挡住了正在编辑的代码区域。

事实上，这个小工具实现原理很简单：使用FindWindow找到任务栏的窗口，将其设置成隐藏的便可以了。
可是整个实现的过程却走了些弯路：

  • 一开始便是错误的使用了FindWindow这个方法。我是通过类名来查找窗口的，所以不用传入Title参数。
    于是我便传进了一个空字符串，而正确的做法是传入NULL。看来最近在搞脚本开发对于C++产生了影响了，是两种语言的编程思想有所差异的原因吧。

  • 再者是自己对于Windows的了解太少了。任务栏隐藏后，程序最大化时也无法占据该区域。
    我一直想通过设置Windows默认窗体最大化的高度来解决这个问题。可是我没有认识到，这个值Windows肯定是不可能直接暴露出来的。
    后面 @王俊 同学提醒到，这和全屏功能很像，并找出了问题根本的原因————没有捕获其他程序的 WM_GETMINMAXINFO的消息。

  • 最终，想到为什么不先将任务栏设置成自动隐藏，如果出现了开始所说的情况再将任务栏设置成不可见呢。

为了方便操作，在实际的使用中我用http://mayhem.codeplex.com/这个工具来绑定了快捷键。
昨天刚在新闻里面看到Mayhem，使用它主要原因是为了尝鲜。

工具源码的地址：https://github.com/kkfnui/Experiments/tree/master/My%20Tools/Control_Task_Bar

现在这样设置之后，看任务确实不方便了。有时想切换任务不方便，尤其是用Window默认的Alt+Tab操作。于是就在网上找了个工具：http://www.ntwind.com/software/vistaswitcher.html
