SDK接口胡言乱语

发表于 2014-06-02   |  

背景

现在在做新版的下载库sdk，最初的p2s功能基本完工，sdk的整个框架都已经确定了。最近要做sdk一个核心的需求——会员增值服务。为了满足这个需求，sdk需要提供新的接口。

看了下目前sdk对外暴露接口的方式，实现简介高效。但是也有个问题，扩展性不高。所以需要修改暴露接口的方式。

接口要求

sdk的接口对外暴露下载资源的功能。调用者可以调用接口创建下载任务，获取任务状态信息，停止任务等。

在我看来，sdk对外暴露接口有下面几个要求：

  1. 接口调用的安全性，对于非法调用不能出现未定义错误
    1. 参数的合法性检测
    2. 接口是有状态的，调用顺序的合法性检测
  2. 除初始化反初始化接口，其他接口需要保证线程安全

sdk工作线程

sdk的工作(主)线程是sdk初始化时创建的。也思考过，为什么不在调用者的线程上工作呢？可能问题比较傻，但还是对比下好处吧：

  1. 实现逻辑独立，可控性强
  2. 接口设计简单，不依赖与调用者的实现通用性强
  3. 反之，如果调用者是多线程调用，则会傻眼

解决方法

参数合法性检测

以上接口的设计已经体现了参数的合法性检测了。创建任务的接口简单的实现方式：

对外直接暴露任务指针也就意味着，后面的接口会使用外部传入的指针。如果传入的指针是非法的，最终sdk会访问到非法地址。

而目前的接口设计，对外暴露的仅有任务句柄。内部保存了task_id和task对象的映射关系，如果传入的task_id未出现在映射关系表中，即认为调用非法。

映射关系需要维护的——创建任务时增加映射关系，释放任务时移除映射关系。之后多线程调用接口就必须给操作映射关系的相关代码加锁了。

调用顺序的合法性

接口的调用最终都需要到主线程中执行，即需要线程间通信。线程间通信的实质是数据共享。
一般线程间通信都是采用消息队列的方式实现的。原有的实现并没有采用这种做法。使用了最淳朴，最实在的数据共享。

每种任务状态有一个队列。如：

  • 需要开始的任务——waiting_start_list
  • 正在运行的任务——running_list
  • 需要释放的任务——waiting_stop_list

接口线程调用start_task时,在接口线程将任务存于waiting_start_list之中。待主线程的循环执行到响应接口线程的时候，便开始任务，并将任务移到running_list之中。其他操作同理。

这样的设计，可以很好的满足创建开始、查询、停止释放的接口需求。但是在需要增加新接口就不方便了。比如现在要增加一个修改任务属性的接口，就不知道怎么添加了。如果是增加一个类似的队列，那么不同操作之前的顺序怎么保证？

备注：使用多个队列来保存任务操作可以解决一个问题，合并任务操作。在测试程序的时候，经常会频繁的点击开始和暂停按钮。假如创建一个任务之后，用户在主线程循环这个时间内调用了4次开始、停止任务。主线程一次性获取到这些操作后，可以现在不同的队列对任务进行切换，最终的操作依赖与队列中的内容。

为了程序的扩展性，这里最终使用消息队列的方式实现。

这样所有的操作都放在了同一个队列之中，先后顺序可以在队列中体现出来。另外，扩展性也较好，增加新的接口仅需继承Command即可。

锁

前面提到多线程通信的本质是数据共享。数据共享必然要解决读写冲突的问题，这里使用锁来保证读和写操作的唯一。

接口总的说来有3类锁：

  1. 任务id和任务对象映射的锁，接口线程存在多线程
  2. 命令队列的锁。多个接口线程，接口线程和主线程的读写
  3. 任务的状态的锁。

目前query_task的接口实现较为特殊,操作没有添加到命令 队列中。这样做出于两点考虑：

  1. query_task调用较频繁
  2. query_task是唯一一个获取任务数据的接口，即有返回值

所以，每个任务的任务信息都需要单独的一个锁。

其他

状态与异步/同步操作

对一个有状态的对象操作。

异步操作

  1. 执行结果唯一，调用后立即修改状态
  2. 执行结果不唯一
    1. 改成pending状态，其他对对象的操作无效
    2. 改成同步操作，在该调用返回前其他操作不可调用。

状态的保存

  1. 对象内部
  2. 列表扩展

HttpDataPipe中的内存管理

发表于 2014-05-17   |  

在实现HttpDataPipe的过程中，在内存管理候花了比较大的精力。由于没有把问题分析清楚，内存管理的逻辑修改过好几次。想找个人分享，但是自己想清楚了总觉得没啥内容。这个算是自己的一个总结把。

问题描述

在我现在做的模块中对于内存管理这里有几个问题需要解决：

  1. 存在同步释放和异步释放的可能。底层的网络库是异步调用的，内存释放必须等网络返回通知;
  2. 内存申请有多种方式。接收数据的内存必须从内存池中分配，有一定的限制；协议数据可以无限制的申请；
  3. 收到的纯数据是不能在Pipe中模块释放的。同上为了减少内存拷贝

网络层接口

一个典型的异步接口。调用的时候需要申请内存，传递给网络层用来存储从网络接受到的数据。

  • s， 套接字句柄
  • pBuffer, 传递给网络层的缓冲区
  • len， 缓冲区长度
  • callback，回掉函数
  • userdata，回掉函数回掉时带的参数
  • opid，本次操作的标志，可以通过该id取消异步操作

内存管理编码的过程

总体来说，内存管理的逻辑有三个版本：

  1. 在Pipe中管理内存，使用BufferAllocRecord记录相应的情况
  2. 使用operation来抽象网络操作，内存在operation中管理
  3. 改变pipe和operation的对应关系，由1:1 —> 1:n (n>=0)

一般异步释放资源的思路，先记录资源，等异步返回的时候从记录中获取资源信息并释放。

封装operation管理内存

在HttpDataPipe整个流程写完了之后，我发现了几个问题：

  1. 每次网络操作都需要申请内存，生成BufferAllocRecord
  2. 调用申请内存的接口不一样
  3. 整个HttpDataPipe流程下来，网络操作不存在并发的问题，即如果有网络操作要么是在读要么就是在写

发现这些问题后，且受PC下载库的影响，就将Record扩展成了一个Operation。Operation除了记录Record的信息外，还提供了管理内存的功能。

另外对HttpDataPipe也做了调整，必须继承接口AsynIoHandler:

“这样代码就存在复用的价值了，以后在实现FtpDataPipe的时候就可以直接使用这套逻辑了”，当时做的时候得意的想到。

整体的接口就是：

  • HttpDataPipe继承AsynIoHandler，依赖AsynIoOperation
  • AsynIoOperation依赖AsynIoHandler

由于发现HttpDataPipe的工作模式是半双工的，所以认为可以提高性能——减少Operation对象生成和销毁的次数。于是就在HttpDataPipe中增加一个Operation的成员变量。

从现在看来这个方案的改动是有问题的，比第一个方案还差劲。问题表现在：

  1. 降低了内存申请和释放的灵活性（想这个方案时的潜规则，实际使用的时候是可以突破的）
  2. 仅仅一个Operation，意味着在下一次操作前必须重置Operation的状态。

问题一表现在接受http响应头的处理逻辑。在收到额外raw data时，必须从Operation中AllocBuffer(raw data必须从memory pool中申请，而header的缓冲区是new出来的)拷贝raw data。这里仅仅是内存拷贝，没有涉及到网络操作，却依赖于Operation。这个确实可以直接修改掉，而不需要改动整个方案，但同时方案的概念也有被改动了。

更改pipe和operation的对应关系

重新整理思路，不纠结那些细节，内存最终只有两个结局：

  1. 直接释放
  2. 交给上层释放

如果，再次操作operation不在依赖于之前的operation，即允许pipe同时拥有多个operation，那么也就不需要在分支上立即重置operation。

从现在代码逻辑上讲，异常的分支会多些。后期维护代码也是这样的，正常逻辑分支肯定会更少，而且是最先确定的。所以，可以在调用HandleHeader后默认释放内存，中间有需要交给上层释放，则做下标记即可。

此次改进还对operation重新进行了定义。

  • operation封装网络操作，为了方便内存的控制，其中需要申请和释放内存的方法，但是Pipe中的内存不一定需要通过operation申请。

  • 针对不同的网络操作可以有不同类型的operation，比如接受header和接受body就不同。主要区别于，申请和释放内存的方式不一样，如开始所提到的要求。

《中国游记》读后感——转折的叙述

发表于 2014-05-12   |  

周末茄子把介川龙之介的《中国游记》“塞”给我。薄薄的本，蛮精致，因此一有空闲便拿出来看了。平日里面看的也都是计算机相关的书籍，一个味道，看多了也会让人感觉乏味。

《中国游记》记录的是日本知名作家介川龙之介在1920年左右到中国来采风的所见所闻。我看了个开头，就觉得有点欣喜了。倒不是对文章的内容给我带来了意外的收获，而是做着的叙事手法顺畅。关于文章内容可以看豆瓣的书评：《中国游记》豆瓣。

下面是我摘抄出来文章的头两段。写的是作者晕船这么一回事。

就在即将从东京出发的那天，长野草风来访。他说，它打算半个月之后也去中国旅行。那是长也曾关切地建议我，随身带上防止晕船的灵丹妙药。可我想，从门司上船之后，只要两天两夜就到上海了。充其量不过两天两夜的海上旅行，还劝我带什么晕船药。长野的胆小如鼠，可见一斑。三月二十一日下午，在我攀上筑台丸的舷梯时，看见在风雨中波浪起伏的港湾，再一次对我们这位长野草粉画伯德怯海情结深表同情。

可是正如俗话说所，不听古人言，吃亏在眼前。当船一进入玄海，转瞬之间，大海变得波涛汹涌起来。我和分在同一船室的马杉君一起，坐在船头甲板的藤椅里，只见海浪拍打着船舷，澎湃作响，不时地有水沫飘落在我们的头顶。海面上不用说已是白浪滔天，犹如一锅开水似的上下翻滚，轰然作响。远处有一个模糊不清的岛影进入眼帘，那该是九州的本岛吧。既便如此，坐惯了船的马杉君却还是嘴里衔着香烟，吞云吐雾，悠然自得，全无窘迫之状。我竖起外套的领子，把两手插在口袋里，是不是低送一颗仁丹到嘴里含着。总而言之，直到这时，我才深深佩服长夜草风上船之前准备晕船药的英明之举。

我看了这段之后，也想到自己经常会写一些转折的内容。但是时常给自己的感觉是，转折太过突然。事情还没有描述清楚，便迫不及待地想将结果告诉大家。一下子也想不到具体的例子。如果让我来写晕船这个事情，估计就下面这点内容吧。

此次去上海前，长野让我带点晕船药。我想大可不必，也不过是两天的航程。也不知道长野为什么这么胆小。结果往往出乎意料，到了船上我很快就开始晕船了。

自己看着，总觉得有点像是一个冷笑话——从前有个小朋友，他不听妈妈话，结果死了。对于叙事我像是不太喜欢，估计是沾上了“嫌麻烦，写不好”的问题。

分析介川写的内容，应该分了几个部分：

  1. 起、事情的源头
  2. 过程描述
  3. 自己的感想

对于转折这块，也对环境大肆的描写了一番。这样引出晕船这个事实，就显得很自然。最后，还不忘写下自己的心理活动，倒像是和人对话般亲切。

被评价：编写高质量代码

发表于 2014-05-09   |  

考评评语

最近同事给了我一句评语“编写高质量代码”。我很欣慰也很兴奋。欣慰的是自己被认同，兴奋的是自己关注的点被认同。

之前收到的一些评语，基本没有让我感到兴奋。评语内容一般是——工作态度积极、责任心强、能力强。这些评语一来太宽泛了，还有这些方面我真的不太关心。

过去半年

不过从某种程度来说，这和之前的工作内容有关系。在过去的半年里，我是负责PC端云加速开放平台。对于开发来说，主要的工作是bug修复和业务需求，实际编写一个模块的机会是很少的。所以主要的工作目标：

  1. 尽快的修复bug、完成需求，做好产品的质量
  2. 因为维护代码碰到的是坑，所以希望写的代码可维护性强

实际在工作中解决问题的时，就会碰到修复bug需要考虑非常多的逻辑，结果修复一个bug的周期很长。最后给自己的挫折感还是蛮强的。一个插曲：去年和邢老大开会，本来是想吐苦水说项目维护难，结果表达不恰当 给理解成了不想做。

由于工作中的挫折感，外加自己对系统和算法都没有实在的了解(知道有这些这些点,但细节不了解)，我自己还是觉得自己能力是不行的。过去一直是在梳理模块内的逻辑，确实对于底层和算法都没有深刻的认识，这两点是需要补充的。

就在这种自我感觉不好的情况下，得到了一句这么实在的评语真的很兴奋。

编码的一些总结

最后，说下我在新项目中写代码的一个基本过程：

  1. 对于整个项目整体框架的设计和理解。都是下载库，变动不是很大
  2. 设计需要实现模块与外界接口和通信流程
  3. 实现最基本的逻辑
  4. 在之前的基础上加功能
  5. 梳理增加功能后的逻辑重构
  6. 步骤4、5迭代，累加新的功能

要保证代码质量好步骤4、5很重要。

另外，就是一些防止出现坏代码的“条条框框”了。如：

  1. 代码的对称性，有始有终；
  2. 可读性，运行起来要和代码写的一样；
  3. 代码重复，有重复就很有可能代码结构上存在问题；
  4. 断言编程，按照设计的逻辑断言不会出现某些情况

更多框框可以看下《程序员修炼之道》、《Write Clean Code》。

工作中的文档

发表于 2014-01-11   |  

瞎扯(可略过)

文档是记录信息的载体。具有持久的特性，读者不用担心文档会消失，可以随时查看。

人脑的记忆是有限的，所以我们可以借由文档来帮助我们记录那些重要的信息。

工作中文档又是信息的适配器。文档不只是给写的人看的，它还需要为其他人“服务”，有传播信息的作用。读者的拥有的背景知识又不尽相同，所以文档并不只是简单的纪实。需要作者对信息进行抽象、加工以及组织。

文档的好处

为了更好更快的完成工作，我们需要编写一些文档。目前工作中文档带来的好处：

  • 节省交流时间

    在实现具体需求的时候，开发和测试之间需要信息同步。如果不通过文档，而是口头交流，会存在两个问题：

    1. 具体细节多的时候，口头交流的时候可能会遗漏信息

    2. 存在反复交流的可能，比如中间有人忘了部分信息

      使用文档记录信息，开发过程中的信息都可以记录下，之后如果有人忘了还可以看文档，而不需要重复交流。

  • 文档更抽象，有利于整理思路

    这点对开发人员来说是非常好的。埋头写代码时，需要关注的细节比较多，有些时候返回会在主要的逻辑上出些错误。如果能再通过一个文档整理逻辑，就会发现实现中的一些问题。(这里说的是文档时实现后的文档了)

工作中常用的文档

  • 需求文档

    需求文档重要的功能是说明要做什么。我所在项目中，过去的多数需求没有文档。没有需求文档问题也不大，因为需求已经是非常细化的，有些时候都可以怀疑它不是需求而是解决方案。不过我觉得以后最好加上，记录我们做过哪些需求。(一个软件需求的积累是有价值的)

    需求文档大概要哪些信息：标题、背景、说明、完成标准。

  • 设计文档 & 详细设计文档

    这两种文档的界限不是很明确，受需求的复杂度的影响比较大。需求很简单时，设计文档也就是详细设计文档。需求复杂时，两者就有不同的分工了。设计文档偏向于解决问题的方向，而详细设计文档偏向于解决问题的过程。在开发后面需要进行的测试是更关注过程，所以详细设计文档通常是和测试交流的重点。

  • 测试用例

    测试先对解决问题的过程进行分析，记录其中关键的点。并且增加相应的测试条件与预期结果。可以将思考和行动独立成两个步骤。

  • “产品”使用文档

    这里指的产品是泛指，它可以是一个软件，也可以是软件中的一个组件。这是我做开发平台认识比较深刻的一个东西。基于产品的在开发非常依赖于使用文档。这里的文档就不单单是文档了，可能还需要包括使用示例。

我非常愿意将这些文档划分成两类：需求相关文档和基础文档。基础文档和需求文档也并非完全独立的，比如一个需求完全就可以使增加一个接口。这种情况下同一份文档既可以看作是需求相关文档，可以看作是基础文档。

基础文档

上面提到的前三种都是需求相关的文档，说的比较具体了。最后一个是产品基础文档，看着觉得数量比较少，其实也可能有蛮多文档的。基础文档是一个嵌套形式的，比如一个产品有A B C D四个模块，最后应该就有5份基础文档了。

现在，项目中需求相关文档都还不错，流程决定了这个文档必不可少。但是基础文档就受大家的重视了，就像公司一样“看重业务，而不注重基础技术”。

基础文档一般是接口文档或者是协议文档，以及领域中定理。

我们模块的接口文档是没有的，同时接口注释也没有。别说注释可能会和代码表达的意思不同，造成后人的误解。接口的含义如果经常变动那问题才更大。有份好的接口文档，对于使用者来说帮助很大。但是怎么写我也没有太多建议，想到是个好的编写方式————将内容写在程序中，然后通过工具生成文档。

协议文档还是蛮全的。这里可以总结下以前的工作现象————不与分工不同的人交流时，我们是就不写文档。

领域中的定理，这是软件构件的基础。这个必须可以有文档，不知道对刚接触该领域的人帮助多大！

使用redmine进行文档的管理

发表于 2014-01-11   |  

背景

最近一年来接触过3个项目流程相关工具了：trello、禅道、redmine。

  • trello的功能比较简单纯粹。是一个团队协作的工具，方便成员之间同步信息的。

  • 禅道是一个完整的软件项目管理工具。用过一段时间，禅道的项目管理理念我是挺接受的，就是后面的需求再细分任务太麻烦了。后面因为产品和测试都不使用这个东西，也就放弃使用了。里面的主要概念：

    • 产品，比如算是迅雷7吧
    • 项目，迅雷7中可以独立出来的，如会员的插件
    • 版本，这个版本我的理解是项目的版本
    • 需求
    • 任务，这个是对需求按照角色再具体细化的工作了

  • 使用redmine是基本上算是被迫的了，没得选择。在放弃使用禅道后，有重新使用trello。但这个东西容易被墙，大家使用的时候也不是很顺畅。刚接触redmine的时候被里面的一个功能深深吸引了——wiki。但是接触过一段时间redmine后，发现用户体验真实渣渣。redmine里面的文字真的不是给人看的——密密麻麻的。另外，目前对于redmine的使用也有点问题，将子项目当作版本，与redmine的管理理念不符。

总体来说，按用户体验来排序 trello > 禅道 > redmine。用户体验太差，这也是为什么在今天之前不能接受redmine的原因。之前作为一个纯使用者，确实没有怎么了解这个工具。今天看了下redmine官网，发现它自己也是基于redmine搭建的，感觉他们的使用还可以。稍微研究了下，这东西还是个开源的软件，支持插件也可以更换主题，也有一定的社区支持。

总之，现在我对于redmine还是抱有一点希望的。接下来我想说下我关于使用redmine进行文档的管理的想法。

怎么会想到使用redmine管理文档

也许有人会问现在管理文档的方式不行吗？可以，但是现在使用word编写文档，按照时间创建目录结构的方法有些缺点可以被改善。

在做下载库之前，做需求的时候不需要写各种文档。在接触下载库之后，开始需要写些文档，参考些文档。按照前人的习惯，就是用word来写文档。做过一两个需求之后，觉得有些问题不是很方便。

  • 首先，我要参考以前的文档，找文档是一件很难的事，不可以通过关键字找。相比于之前做界面开发这个就麻烦多了，以前可以使用google查找到新知识的帮助信息。
  • 其次，也许是个人因素，比较认同web的文档。web文档更容易体现知识的体系结构，这点得益于web方便的链接跳转功能(word应该也可以，但是大多数人不会使用吧)。
  • 最后，使用word写文档，格式不统一也是一个非常明显的问题，就我自己写过的word文档的格式已经是千差万别了。当然这个不是最重要的，但是如果可以改善就最好了。

现在下载库有很多文档，但是由于检索的不便，这些文档很少会被人阅读到。不能充分发挥文档的作用。redmine中有全局搜索功能，这是一个比较好的索引功能，可以为各个文档带来流量体现它们的价值。

redmine在文档管理上的缺点

刚刚说了现在的缺点，准确说这些缺点是针对redmine可以解决的点上列出来的。redmine本身也是有问题。redmine的文档管理主要是基于wiki的。wiki对于我们有下面几个缺点：

  • 使用wiki编写文档需要记一套特有的语法，需要团队额外的学习成本
  • 对富文本编辑的支持不足，详细设计文档中我们经常画些流程图，wiki中添加图片的过程就比较麻烦了
  • 旧文档的迁移

也有人可能会说，目前的文档管理方式数据最终放在svn上的，数据很安全安全。而redmine的数据不安全。这个事实我得先承认，但是这个不是redmine的固有缺点，而是对于redmine我们没有提供更多的保障措施。

redmine在文档管理上有优点也缺点，但是利大于弊。优点带来的好处意义深远，而缺点带来的痛苦是短暂的，也是可以被优化的。

怎么做

我对于项目中文档的理解：工作中的文档

目前对项目中文档可分为两类：需求相关文档和项目基础文档。redmine中的两个主要功能可以与之一一对应。将需求相关的文档写在问题中，基础文档写在wiki中。

基础文档可以补充的内容：

  • 项目介绍（对内真的可以不用，但是对外对老大还是有用的）
  • 源码及编译方法（对新加入的开发特有用）
  • 领域内的基本概念

以上是基于一个开发的角度考虑的

对于怎么做，目前也是没有更多经验。基本也就靠摸石头过河吧。

redmine的wiki语法参考：RedmineTextFormatting

延伸

工具的使用的深度和工具带来的效益的关系是一个倒抛物线的。工具的使用浅尝辄止和过度依赖都不是最好的，对于redmine这个工具我们还可以再挖掘下，而不是纯粹的用于记录。

  • Redmine的样式：http://redminecrm.com/projects/themes/pages/1
  • Redmine手册：http://ishare.iask.sina.com.cn/download/explain.php?fileid=37418817

svn分支开发流程说明

发表于 2014-01-03   |  

工作中我们使用svn来管理源码。之前基于svn来开发需求的基本流程：

  1. 开发在本地将需求完成
  2. 将代码修改提交到svn，由编译机统一编译
  3. 统一编译后，将产品提交给测试
  4. 开始测试产品

这样的使用方法会有以下几个问题：

  1. 不能及时的将代码提交到svn上，存在代码丢失的风险

  2. 需求内bug的修复，提交svn产生噪音

    上面开发的基本流程，在最理想的状态下是没有问题的。事实上，步骤2，4都会产生问题,造成主干上的垃圾日志就变多。

    • 提交时忘记先添加某个新加的文件(开发本地编译没有问题，但是编译机无发通过)
    • 测试发现问题，这个时候必须修复问题吧，又得提交一次

  3. 不方便bug定位

    这点也是因为上面说的日志噪音产生的。想象下产品上线后，发现一个bug，我们需要定位是哪个需求引入的。
    一个需求的svn提交记录很多，到底改了哪些东西就不是那么容易确定了。

svn使用流程

对于上面提出的问题，解决方案会有多个。可以要求开发人员提交svn时不能出现低级错误，开发需求不能出现bug。好吧，这个是开玩笑的了。从现在的经验看来，这些问题都无法避免。错误既然无法避免，那么就修改流程允许错误的出现，同时不会带来坏的影响。

下面就是我关于svn使用流程的一个想法：

  1. 一个需求必须创建一个独立的svn分支
  2. 开发和测试这个过程，都是基于这个svn分支的
  3. 测试确认最终没有问题后，合并到主干上

在理想情况下，主干上的每次修改都是稳定的、可发布的。

同上面分析以前的流程一样，我们同样不能将整个流程看的太顺利。

  1. 创建svn分支。

    不会创建svn分支？不能创建svn分支？如果在这里出现了问题，影响将会是最小的

  2. 开发和测试的迭代

    这个过程是允许出错的

  3. 合并需求到主干

    这个过程很重要，这就相当于一个产品的发布。在没有修改冲突的时候，我们可以直接使用svn的merge操作。比较麻烦的是有冲突，需要相关人员一起来合并代码了。

问题

  1. 每个需求都创建一个分支是否有必要？

    有必要。

    如果不是一个需求一个分支，那么两种可能：多个需求多个分支和一个需求多个分支（有病的感觉，纯粹是分析问题为了保证完整性）；多个需求一个分支，这个挺正常的。

    先来分析下最后一种情况，说下这样做不太合适的地方。合并一个分支到主干发布，这个分支包括了需求1、2、3。现在产品发布出去了，结果出现了bug（性能问题）。现在我们要分析是那个改动引入的。分析定位问题要做到，最终问题可以被，所以越精确越好。这种情况逐渐缩小搜索问题可能出现的范围是一个最好的解决方案。即需要定位哪个需求引入的，那么要从一次提交中拆分三个不同的需求。没有做过需求的人根本不知道怎么拆分。

    每个需求一个分支除了能解决上面的问题，还允许并行开发。

    最后，我有一个目标，就是在主干的日志上，可以一目了然的看清楚在什么时间发布了什么改动。

  2. 很简单的一个需求，创建一个分支，最后还需要合并代码是不是过于麻烦了？

    在回答这个问题前，我希望我们对于第一个问题已经达成共识。

    简单的定义，这是一个比较泛的东西，不容易界定。这里就给它一个定义：与其他需求在代码的修改上不存在冲突。现在这个问题的关键字就是麻烦了。我不认为这是一个麻烦的操作，不存在冲突，合并代码这个操作工具就可以帮助完成了。

工具的使用

工具不是我们的目标，但是工具可以帮助我们实现目标，影响我们的思路。

下面我说下我对svn分支操作的理解。工作中都是使用Tortoise SVN客户端的，下面就使用它来说明下相关的操作。

  • 如果svn基本操作不会的可以先参考：http://blog.csdn.net/dily3825002/article/details/6717076
  • 更多更具体的操作可以参考：http://tortoisesvn.net/docs/nightly/TortoiseSVN_zh_CN/index.html

按照上面说的流程，需要创建分支、在分支上开发、合并分支。

创建分支比较简单，可以参考上面的两个链接。

创建完成分支之后，需要在分支上开发。以前我比较笨，就从svn的上checkout分支下来。其实不需要这么麻烦，svn有个switch命令，可以改变本地路径对应的svn url。给人的感觉，创建分支是和没有创建分支一样。详细的介绍：http://www.cnblogs.com/dabaopku/archive/2011/05/21/2052820.html

分支合并有两种情况，没有冲突和有冲突。

Phrase Emphasis

*italic*   **bold**
_italic_   __bold__

Links

Inline:

An [example](http://url.com/ "Title")

Reference-style labels (titles are optional):

An [example][id]. Then, anywhere
else in the doc, define the link:

  [id]: http://example.com/  "Title"

Images

Inline (titles are optional):

![alt text](/path/img.jpg "Title")

Reference-style:

![alt text][id]

[id]: /url/to/img.jpg "Title"

Headers

Setext-style:

Header 1
========

Header 2
--------

atx-style (closing #’s are optional):

# Header 1 #

## Header 2 ##

###### Header 6

Lists

Ordered, without paragraphs:

1.  Foo
2.  Bar

Unordered, with paragraphs:

*   A list item.

    With multiple paragraphs.

*   Bar

You can nest them:

*   Abacus
    * answer
*   Bubbles
    1.  bunk
    2.  bupkis
        * BELITTLER
    3. burper
*   Cunning

Blockquotes

> Email-style angle brackets
> are used for blockquotes.

> > And, they can be nested.

> #### Headers in blockquotes
> 
> * You can quote a list.
> * Etc.

Code Spans

`<code>` spans are delimited
by backticks.

You can include literal backticks
like `` `this` ``.

Preformatted Code Blocks

Indent every line of a code block by at least 4 spaces or 1 tab.

This is a normal paragraph.

    This is a preformatted
    code block.

Horizontal Rules

Three or more dashes or asterisks:

---

* * *

- - - -

Manual Line Breaks

End a line with two or more spaces:

Roses are red,   
Violets are blue.

解析markdown的想法

发表于 2013-12-27   |  

前因

这几天用Markdown写东西越写越high了。写文主要的是内容，但是Markdown的体验真的很不错。网上有各种Markdown的编辑器，各有各的特点，功能基本满足我的需求。

StackEdit 这个在线编辑器功能很强大，用户体验很好,这篇文档就在这上边编写完成的。它提供了和google、dropbox以及github的互操作，对我来说的功能已经溢出了。

前几天使用的是马克飞象 这款编辑器写的文档，可以和Evernote打通，也非常棒。比较两者，发现各自对Markdown的语法支持有差异。

使用两者之后的感受，各有各的好处吧。但是这种面向大众使用产品，个人的操作效率肯定不是最高的。比如，我特别想通过StackEdit一键发布博文，保存到GitHub的路径可以自动确定。有种声音，改变它，让它符合自己的操作习惯。

上面提到的StackEdit编辑器是开源的，如果想增加特化的功能，可以自己修改源码实现。源码地址：https://github.com/benweet/stackedit

从满足自己的需求角度出发，我应该基于StackEdit自己再做二次开发的。可是总有一种想法驱赶我:”你应该自己造个轮子解析Markdown”。
这两个事相差的有点远了。有些时候，也不知道自己为什么会这样想。

应该就是我能这个道理吧。

一点想法

词法解析这块完全没有经验。

印象中需要使用到树和状态机吧。大学里面没有好好学习数据结构，学了的也还给老师了。所以这块肯定是要重现学习的。状态机倒也还算实践过，但却也没有说能轻松搞定的勇气。

涉及到字符串编码的问题，这个倒还比较熟悉。却也没有真正研究过，比如在内存解析时使用utf8编码，怎么遍历字符和字符匹配呢？

总体来说，是零基础开始做了。

计划

这块肯定已经有比较成熟的解析器。应该先借鉴下现有的实现，怎么也得踩下别人的肩膀。另外现在已经可以预见的知识也要准备下：

  • 树数据结构算法
  • 状态机
  • 字符串编码
  • Markdown的语法

最后一句，”善始者实繁，克终者盖寡”。算是激励自己，也为自己找个台阶。

补充：突然想到自己做事不从需求出发，应该是被人洗脑了，看大神言论看多了。算了，这次认栽了！
