Header-Of-Line-Blocking

发表于 2015-03-25   |  

在了解HTTP/2的时候看到了这个名词，字面意思是在队列前面的发生了阻塞。

在维基百科里面了解到了以下情况会出现：

  • 使用FIFO 输入缓冲队列的交换机，改现象会大大降低交换机的吞吐量
  • HTTP pipeline

两者的共性是相同的，前面有个比较弱的家伙在挡道。

SWITCH

交换机有4个输入队列，4个输出端口。第一和第三输入队列都在给第四输出端口发送数据，此时可能先发送第三队列的数据。那么第一个队就要阻塞，而其后面的一个数据是要发送到第三输出端口。造成资源浪费了。

HTTP

在HTTP 1.0 的时候，每个连接只能请求一个资源。这种方式性能较低。建立连接成本高，而下载网页或者小图片其传递的数据量不多。
后来在 HTTP 1.1时，引入了 keepalive、 pipeline。

  • keepalive，允许连接复用从而更提升连接的利用率，但是增加了第二次请求响应的延迟
  • pipeline，允许在第一次响应完全接收到之前可以发送第二次请求，从而减少延迟

在使用pipeline的时候就会碰到了HOLB的问题，比如第一次请求的是一个大文件，而后面请求的文件较小。那么小的文件就会被阻塞了。

How HTTP/2 Addresses HOLB

HTTP/2 通过多路复用解决了这个问题。在一条TCP连接上可以同时进行多次请求和响应。这是怎么做到的呢？TCP不是可靠有序的传输吗？

粗略的看了下HTTP/2在TCP的传输层之上重新抽象了一个传输层。 TCP流—> Frame —> 虚拟流。
