HttpDataPipe中的内存管理

发表于 2014-05-17   |  

在实现HttpDataPipe的过程中，在内存管理候花了比较大的精力。由于没有把问题分析清楚，内存管理的逻辑修改过好几次。想找个人分享，但是自己想清楚了总觉得没啥内容。这个算是自己的一个总结把。

问题描述

在我现在做的模块中对于内存管理这里有几个问题需要解决：

  1. 存在同步释放和异步释放的可能。底层的网络库是异步调用的，内存释放必须等网络返回通知;
  2. 内存申请有多种方式。接收数据的内存必须从内存池中分配，有一定的限制；协议数据可以无限制的申请；
  3. 收到的纯数据是不能在Pipe中模块释放的。同上为了减少内存拷贝

网络层接口

一个典型的异步接口。调用的时候需要申请内存，传递给网络层用来存储从网络接受到的数据。

  • s， 套接字句柄
  • pBuffer, 传递给网络层的缓冲区
  • len， 缓冲区长度
  • callback，回掉函数
  • userdata，回掉函数回掉时带的参数
  • opid，本次操作的标志，可以通过该id取消异步操作

内存管理编码的过程

总体来说，内存管理的逻辑有三个版本：

  1. 在Pipe中管理内存，使用BufferAllocRecord记录相应的情况
  2. 使用operation来抽象网络操作，内存在operation中管理
  3. 改变pipe和operation的对应关系，由1:1 —> 1:n (n>=0)

封装operation管理内存

在HttpDataPipe整个流程写完了之后，我发现了几个问题：

  1. 每次网络操作都需要申请内存，生成BufferAllocRecord
  2. 调用申请内存的接口不一样
  3. 整个HttpDataPipe流程下来，网络操作不存在并发的问题，即如果有网络操作要么是在读要么就是在写

发现这些问题后，且受PC下载库的影响，就将Record扩展成了一个Operation。Operation除了记录Record的信息外，还提供了管理内存的功能。

另外对HttpDataPipe也做了调整，必须继承接口AsynIoHandler:

“这样代码就存在复用的价值了，以后在实现FtpDataPipe的时候就可以直接使用这套逻辑了”，当时做的时候得意的想到。

整体的接口就是：

  • HttpDataPipe继承AsynIoHandler，依赖AsynIoOperation
  • AsynIoOperation依赖AsynIoHandler

由于发现HttpDataPipe的工作模式是半双工的，所以认为可以提高性能——减少Operation对象生成和销毁的次数。于是就在HttpDataPipe中增加一个Operation的成员变量。

从现在看来这个方案的改动是有问题的，比第一个方案还差劲。问题表现在：

  1. 降低了内存申请和释放的灵活性（想这个方案时的潜规则，实际使用的时候是可以突破的）
  2. 仅仅一个Operation，意味着在下一次操作前必须重置Operation的状态。

问题一表现在接受http响应头的处理逻辑。在收到额外raw data时，必须从Operation中AllocBuffer(raw data必须从memory pool中申请，而header的缓冲区是new出来的)拷贝raw data。这里仅仅是内存拷贝，没有涉及到网络操作，却依赖于Operation。这个确实可以直接修改掉，而不需要改动整个方案，但同时方案的概念也有被改动了。

更改pipe和operation的对应关系

重新整理思路，不纠结那些细节，内存最终只有两个结局：

  1. 直接释放
  2. 交给上层释放

如果，再次操作operation不在依赖于之前的operation，即允许pipe同时拥有多个operation，那么也就不需要在分支上立即重置operation。

从现在代码逻辑上讲，异常的分支会多些。后期维护代码也是这样的，正常逻辑分支肯定会更少，而且是最先确定的。所以，可以在调用HandleHeader后默认释放内存，中间有需要交给上层释放，则做下标记即可。

此次改进还对operation重新进行了定义。

  • operation封装网络操作，为了方便内存的控制，其中需要申请和释放内存的方法，但是Pipe中的内存不一定需要通过operation申请。

  • 针对不同的网络操作可以有不同类型的operation，比如接受header和接受body就不同。主要区别于，申请和释放内存的方式不一样，如开始所提到的要求。
