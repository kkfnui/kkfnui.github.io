被评价：编写高质量代码

发表于 2014-05-09   |  

考评评语

最近同事给了我一句评语“编写高质量代码”。我很欣慰也很兴奋。欣慰的是自己被认同，兴奋的是自己关注的点被认同。

之前收到的一些评语，基本没有让我感到兴奋。评语内容一般是——工作态度积极、责任心强、能力强。这些评语一来太宽泛了，还有这些方面我真的不太关心。

过去半年

不过从某种程度来说，这和之前的工作内容有关系。在过去的半年里，我是负责PC端云加速开放平台。对于开发来说，主要的工作是bug修复和业务需求，实际编写一个模块的机会是很少的。所以主要的工作目标：

  1. 尽快的修复bug、完成需求，做好产品的质量
  2. 因为维护代码碰到的是坑，所以希望写的代码可维护性强

实际在工作中解决问题的时，就会碰到修复bug需要考虑非常多的逻辑，结果修复一个bug的周期很长。最后给自己的挫折感还是蛮强的。一个插曲：去年和邢老大开会，本来是想吐苦水说项目维护难，结果表达不恰当 给理解成了不想做。

由于工作中的挫折感，外加自己对系统和算法都没有实在的了解(知道有这些这些点,但细节不了解)，我自己还是觉得自己能力是不行的。过去一直是在梳理模块内的逻辑，确实对于底层和算法都没有深刻的认识，这两点是需要补充的。

就在这种自我感觉不好的情况下，得到了一句这么实在的评语真的很兴奋。

编码的一些总结

最后，说下我在新项目中写代码的一个基本过程：

  1. 对于整个项目整体框架的设计和理解。都是下载库，变动不是很大
  2. 设计需要实现模块与外界接口和通信流程
  3. 实现最基本的逻辑
  4. 在之前的基础上加功能
  5. 梳理增加功能后的逻辑重构
  6. 步骤4、5迭代，累加新的功能

要保证代码质量好步骤4、5很重要。

另外，就是一些防止出现坏代码的“条条框框”了。如：

  1. 代码的对称性，有始有终；
  2. 可读性，运行起来要和代码写的一样；
  3. 代码重复，有重复就很有可能代码结构上存在问题；
  4. 断言编程，按照设计的逻辑断言不会出现某些情况

更多框框可以看下《程序员修炼之道》、《Write Clean Code》。
